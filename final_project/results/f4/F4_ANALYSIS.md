# F4 — frozen full test + Pi benchmark (Raspberry Pi CM5)

**Date:** 2026-07-20. **Frozen config:** threads=1, full decode, `ORT_ENABLE_ALL`, arena on,
fused, 256×192 (the F3 freeze; the exact config every P1–P4 gate validated). Latency on the
**real Raspberry Pi CM5** (BCM2712 / Cortex-A76 @ 2.4 GHz, 8 GB, Ubuntu 24.04, performance
governor, no throttling). Test accuracy is gx10 C++/ORT of the frozen bundle artifacts run in
the target container (`accuracy only`, §12.4) — shown to transfer to the Pi bit-identically by
the parity check. All numbers generated by `scripts/summarize_f4.py` → `f4_summary.json`.

## 1. On-device latency — baseline vs optimized (3 separate-process reps, ≥1000 iters each)

| model | kind | end-to-end p50 (mean ±spread) | FPS | peak RSS | speedup |
|---|---|---:|---:|---:|---:|
| **M0** | FP32 baseline | 49.06 ms (±1.14) | 20.4 | 96.5 MB | 1.00× |
| **M2** | INT8 QAT — **FINAL** | 21.61 ms (±0.36) | 46.3 | 89.1 MB | **2.27×** |
| M4 | pruned + QAT INT8 | 17.74 ms (±0.21) | 56.4 | 82.2 MB | 2.77× |

The reps are tight (M2 ±0.36 ms over three fresh processes), so the numbers are stable. The
frozen deliverable is **M0 → M2: 20.4 → 46.3 FPS, a 2.27× on-device speedup at 3.5× smaller
model (8.95 → 2.54 MB)**. M4's 2.77× is reported as the most-aggressive optimization measured
(not selected, F3/§8.4).

## 2. Frozen full-test accuracy (test set opened once, post-freeze, frozen threshold)

Held-out `cis_test` (15,827 frames, 897 bobcat) and `trans_test` (23,275 frames, 2,338 bobcat),
scored at each model's frozen validation-calibrated threshold — never re-tuned on test (§5.4).

| model | split | frame F2 | seq-balanced recall | **event capture** | false-fire |
|---|---|---:|---:|---:|---:|
| M0 FP32 | cis_test | 0.5812 | 0.6636 | 0.7668 | 0.0551 |
| M0 FP32 | trans_test | 0.2517 | 0.2919 | 0.3952 | 0.0409 |
| **M2 QAT** | cis_test | **0.6387** | **0.7604** | **0.8575** | 0.0551 |
| **M2 QAT** | trans_test | 0.2209 | 0.2621 | 0.3468 | 0.0215 |

**Reading it honestly.**
- **In-distribution (`cis_test`): M2 beats the FP32 baseline** — F2 0.6387 vs 0.5812, and it
  captures **85.8 % of bobcat visits** (event-capture) vs M0's 76.7 %, at the same 5.5 %
  false-fire. Quantization did not cost accuracy here; if anything QAT regularized slightly.
- **New locations (`trans_test`): both models drop hard** (event capture ~35–40 %), the known
  domain-shift limitation — bobcats at unseen cameras are the hard case. M2 is a touch lower on
  recall than M0 but fires far less often (2.15 % vs 4.09 % false-fire). Neither is production-
  ready for brand-new sites; this is stated as a limitation, not hidden.
- **False-fire on test is ~5.5 % on cis** — just above the 5 % budget the threshold was
  calibrated to *on validation*. Expected for a held-out split (calibration transfer is
  imperfect); reported as-is, not massaged. Consistent with the `recall_floor_infeasible` status
  carried since C3 — this trigger ships an operating point, it does not pass the 90 % recall
  floor, and every artifact says so.

## 3. Pi ↔ gx10 parity — target equivalence

`run-dataset` on the identical bundled parity slice, real Pi vs the frozen gx10 reference (in
the target container), diffed frame by frame (`scripts/compare_pi_parity.py`):

| model | frames | max score Δ | mean Δ | decision disagreements | verdict |
|---|---:|---:|---:|---:|:--:|
| M0 FP32 | 47 | 0.000e+00 | 0.000e+00 | 0 | PASS |
| M2 INT8 | 47 | 0.000e+00 | 0.000e+00 | 0 | PASS |

**Bit-identical, both models** — even the FP32 baseline, which the registered expectation
allowed to drift (float accumulation order differs between the gx10 X925 SVE2/i8mm path and the
Pi A76 NEON path). It does not: ORT's MLAS kernels for this MobileNetV2 produce identical output
across the ISAs, exactly as the E6 QEMU rehearsal predicted. So the frozen gx10 test accuracy
above transfers to the Pi with **zero** decision drift — the target-equivalence claim holds
without reservation (§12.4).
