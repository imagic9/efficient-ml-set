# F2 — Raspberry Pi validation performance profiling

**Host:** Raspberry Pi CM5 (BCM2712 / Cortex-A76 @ 2.4 GHz, 8 GB), Ubuntu Server 24.04.
**Date:** 2026-07-20. **Governor:** `performance` (pinned — a documented, reversible DVFS
control so the model comparison is not confounded by frequency scaling; **not** model tuning).
**Protocol:** `deploy/pi/run_f2_profile.sh` + `run_f2_threads.sh`, warmup 20, 300 iterations,
one validation sample frame (latency is dominated by the fixed-size pipeline; F4 runs the
≥1000-frame distribution). No thermal throttling occurred (`throttled=0x0`, temp ≤ 56 °C).
All numbers are generated by `scripts/summarize_f2.py` into `f2_summary.json` — never hand-typed.

These are **real Pi results** (DESIGN §12.4): the host is certified `is_pi5_a76=true` by
`results/f1/environment.json`, and every benchmark JSON records `looks_like_pi5=true`.

## 1. Baseline shortlist (shipping config: threads 1, full decode, `ORT_ENABLE_ALL`, arena on)

| Model | kind | end-to-end p50 | FPS | inference p50 | inference share | peak RSS |
|---|---|---:|---:|---:|---:|---:|
| **M0** | FP32 baseline | 48.23 ms | 20.7 | 41.21 ms | **85 %** | 96.5 MB |
| **M2** | INT8 QAT | 21.30 ms | 47.0 | 14.22 ms | 67 % | 89.1 MB |
| **M4** | pruned + QAT INT8 | 17.54 ms | 57.0 | 10.34 ms | 59 % | 82.2 MB |

Model optimization alone (all at the shipping threads=1) gives **M2 2.26×** and **M4 2.75×**
over the FP32 baseline, and pruning also cuts peak RSS 96.5 → 82.2 MB.

## 2. The bottleneck, and how it shifts

Decode is a fixed ~6.0 ms and preprocess ~1.0 ms for every model (same JPEG, same 256×192
letterbox). Only **inference** changes:

- FP32 M0: inference is **41.2 ms = 85 %** of the pipeline — the forward pass is the bottleneck.
- INT8+pruned M4: inference collapses to **10.3 ms (4.0× less)**, so decode+preprocess is now
  **~41 %** of the pipeline. **The bottleneck has moved from the model to the JPEG decode path.**

This is why quantization/pruning wins big and why reduced decode was even considered — and why
the *next* latency gain would come from the decode path, not the model (see §5, next steps).

## 3. Thread-scaling matrix (end-to-end p50, ms) — the Pi-specific finding

| model | t1 | t2 | t3 | t4 | best |
|---|---:|---:|---:|---:|:--:|
| M0 | 48.41 | 35.84 | **34.73** | 36.60 | t3 |
| M2 | 21.32 | 20.50 | **20.28** | 21.25 | t3 |
| M4 | 17.28 | 14.33 | **13.81** | 14.42 | t3 |

**`threads=3` is optimal for all three models** on the 4-core A76; `t4` regresses (the 4th
intra-op thread contends with the OS and the decode/preprocess work). This **contradicts E6 on
gx10**, where `threads=4` regressed to 0.88× — on gx10's 20 cores the intra-op overhead exceeded
the gain on this small MobileNetV2, while the Pi has the opposite balance. A genuine
target-hardware result that emulation could not have produced.

Threading helps FP32 most (M0 1.39× t1→t3) because its inference is the largest and most
parallelizable; the INT8 models gain less (M2 1.05×, M4 1.25×) since they are already fast and
the fixed decode overhead is a bigger fraction.

## 4. M0 single-factor knob matrix (off the shipping baseline)

| knob | e2e p50 | speedup vs baseline | decision |
|---|---:|---:|---|
| baseline (all/arena-on/fused/full/t1) | 48.23 ms | 1.00× | shipping |
| `--graph-opt extended` | 49.24 ms | 0.98× | no gain (ALL stays default) |
| `--arena off` | 47.85 ms | 1.01× | within noise (arena stays on) |
| `--preprocess reference` | 50.16 ms | 0.96× | fused is slightly better |
| `--decode half` | 45.99 ms | 1.05× | **REJECTED — accuracy (E6 decode-drift)** |
| `--decode quarter` | 44.55 ms | 1.08× | **REJECTED — accuracy (E6 decode-drift)** |
| `--threads 2/3/4` | 35.84 / 32.54 / 34.90 ms | up to 1.48× | threads=3 adopted (see below) |

The graph/arena/preprocess knobs are all within noise, confirming the shipping defaults. Reduced
decode buys only ~1.05–1.08× on the Pi (decode is ~6 ms of a 17–48 ms pipeline) and was already
rejected at E6 because half/quarter lose real bobcat detections — not worth the target animal.

## 5. Headline result and decomposition

**Best optimized configuration: M4 (pruned + QAT INT8) @ threads=3 = 13.81 ms / 72.4 FPS**,
versus the **FP32 baseline M0 @ threads=1 = 48.41 ms / 20.7 FPS** — a **3.50× end-to-end speedup**,
decomposed cleanly into the two optimization axes:

- **Model** (M0→M4, both threads=1): **2.80×** — INT8 QAT + structured pruning.
- **Inference** (M4 threads 1→3): **1.25×** — intra-op threading on the 4-core A76.

## 6. Decisions carried into F3 (freeze)

- **Shipping knobs confirmed:** fused preprocess, full decode, `ORT_ENABLE_ALL`, arena on.
- **Reduced decode:** stays REJECTED (accuracy, E6).
- **Threads = 3** is optimal for all models. For the **INT8 winner (M2/M4) it is parity-safe** —
  QDQ convolutions accumulate exactly in int32 and are thread-invariant. For **FP32 M0** threads>1
  changes reduction order, so the FP32 **baseline is reported at the frozen shipping default
  threads=1**; the threaded M0 numbers are reported as a measured inference-optimization finding,
  not the frozen baseline artifact.
- **Next steps (report):** with inference no longer dominant on the INT8 models, the remaining
  latency headroom is in the ~6 ms JPEG decode (a lighter decoder or a smaller stored frame), not
  in the model.
